## Purpose

The single orchestrator for the theme's head asset pipeline. Called once per page from `layout/*.liquid`, emits in fixed order: (1) the four core CSS layers concatenated and inlined + `core.js` preloaded as ES module, (2) dynamic styles (font faces + CSS variables) inlined as a second minified block, (3) the ES module import map. Three stages, three concerns kept separate so the inline `<style>` blocks render in dependency order on first paint.

The orchestration is deliberately sequential — `utility--asset-loader` is called multiple times rather than once with a composite signature. Each call's strategy choice is independent (inline-from-file for layers, captured-string-inline for dynamic styles, preload for the JS entry).

## API

No params. Reads its dependencies' globals via their internal Liquid logic (e.g. `utility--css-variables` reads `settings.*` for color schemes / typography / spacing).

## Output shape

Renders three groups of head content, in order:

```html
<!-- 1. Core layers inline + core.js preload -->
<style>{ four layer-*.css files concatenated, then minified }</style>
<link rel="modulepreload" fetchpriority="high" href="…/core.js?…">
<script type="module" src="…/core.js?…"></script>

<!-- 2. Dynamic styles inline -->
<style>{ @font-face declarations + :root + per-scheme rule blocks, minified }</style>

<!-- 3. ES module import map -->
<script type="importmap">
  { "imports": { "@theme/core": "…/core.js?…", … } }
</script>
```

The first `<style>` block is the substrate's render-blocking CSS — no FOUC. The second carries the dynamic content (font faces + variable emission) generated by Liquid at render time. The import map declares the `@theme/<name>` specifiers consumed by `assets/*.js` modules.

## Behavior

- **Per-layer inline because of Shopify's 15360-byte cap.** Liquid's `inline_asset_content` filter caps a single file at 15360 bytes. The four `layer-*.css` files are captured independently and concatenated to clear the cap; the concatenated string routes through `utility--asset-loader (css: 'inline', css_content: <concat>)`, which runs `utility--css-minifier` once over the full concatenation.
- **Inline-then-preload pattern for the JS entry.** `core.js` ships as a normal ES module file (not inline) — its size and the importmap dependency make inlining a poor fit. The `preload` strategy emits both `<link rel="modulepreload">` and `<script type="module">`; the preload starts the fetch with `fetchpriority="high"` ahead of the module tag's execution.
- **Stage 1 emits one `<style>` block.** The four `layer-*.css` files concatenate before passing to the loader. Capturing each individually then concatenating prevents the per-file 15360-byte cap from blocking the load; the resulting inline string is whatever size the four layers add up to.
- **Stage 2 is the Liquid-generated CSS.** `utility--font-face` emits the `@font-face` declarations sourced from the font/typeface metaobjects; `utility--css-variables` emits `:root` and per-scheme rules from settings + design-system metaobjects. The two captures merge into one inline block.
- **Stage 3 import map after stage 2.** Browsers parse the import map before executing the first module. `core.js` is the module that triggers the import map's resolution, so the import map must land before any of the module-resolved imports execute — emitting last in the pipeline still satisfies the constraint because module scripts defer until parsing finishes, and the import map sits inside the same `<head>` parse window.
- **Single-call discipline.** The utility is designed to be called exactly once per page, from `layout/*.liquid`. Multiple calls would emit duplicate `<style>` blocks and duplicate import maps. Defensive guards against duplicate calls are out of scope — the call-site is well-known.

- **Page(s)**: every shipped validation page exercises the orchestrator (every page renders through a layout, every layout calls `utility--core-assets`).
- **API surface** (matrix to exercise):
  - Stage 1: head contains one `<style>` block carrying minified contents of the four `layer-*.css` files concatenated
  - Stage 1: head contains `<link rel="modulepreload">` + `<script type="module">` for `core.js`
  - Stage 2: head contains a second `<style>` block carrying `@font-face` rules + `:root` + per-scheme rules
  - Stage 3: head contains `<script type="importmap">` listing all `@theme/<name>` specifiers
- **Edge cases**:
  - One of the four `layer-*.css` files missing → the missing file's contents render as empty via `utility--inline-asset`; the other three layers still emit
  - `utility--css-variables` empty (no settings configured) → stage 2's `<style>` block is suppressed by the asset-loader's blank-check
  - `utility--import-map` empty list → stage 3 emits nothing
- **Visual showcase**: every page's `<head>` view in DevTools is the showcase. No dedicated page needed.
- **Assertions** (prose; Playwright once installed):
  - `document.querySelectorAll('head > style').length >= 2` on every page (stages 1 and 2)
  - `document.querySelector('head > script[type=importmap]')` returns the import map element
  - `document.querySelector('head > link[rel=modulepreload]')` references `core.js`
  - The first `<style>` block contains substrings drawn from each of `layer-base.css`, `layer-reset.css`, `layer-theme.css`, `layer-utilities.css`

## Out of scope

- **Per-template asset variation.** All templates share the same head pipeline. Per-template specialized asset loading (e.g. extra `assets/<template>.css`) happens at the section level, not at the layout / core-assets level.
- **Stage reordering.** The three stages are fixed in order. Changing the order would break the dependency chain (core layers first → dynamic styles overriding via cascade → import map for module resolution).
- **Defensive multi-call guard.** The utility assumes single-call discipline at the layout level. Adding an idempotency guard is out of scope.
- **Stage 4 onward.** Future-stage additions (e.g. a CSS-Houdini worklet registration, a service-worker registration script) would land as new stages in this orchestrator, not as new utilities at the layout level.
